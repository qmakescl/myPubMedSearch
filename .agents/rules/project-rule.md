## 프로젝트 구조 및 모듈 구성

이 저장소는 현재 `uv`로 관리되는 최소 구성의 Python 3.12 프로젝트입니다.

* `main.py`: 애플리케이션 진입점입니다.
* `pyproject.toml`: 프로젝트 메타데이터, Python 버전 제약 조건, 의존성을 정의합니다.
* `.python-version`: 호환 가능한 환경 관리 도구에서 Python 3.12를 사용하도록 고정합니다.
* `README.md`: 프로젝트 개요 문서입니다. 사용자 대상 워크플로가 추가되면 내용을 확장하세요.
* `.venv/`는 `uv`가 생성하는 디렉터리이며, Git에 커밋하면 안 됩니다.

프로젝트가 커지면 재사용 가능한 코드는 `suppl/`와 같은 패키지에 배치하고, 테스트 코드는 `tests/`에 배치하세요. 데이터셋, 모델 체크포인트, 생성된 출력물은 작고 의도적으로 포함한 fixture가 아닌 이상 소스 디렉터리와 Git에서 제외하세요.

## 빌드, 테스트 및 개발 명령어

환경 및 의존성 관리는 `uv`를 사용하세요.

```bash
uv sync                 # pyproject.toml을 기준으로 .venv 생성/업데이트
uv run python main.py   # 현재 애플리케이션 실행
uv add <package>        # 런타임 의존성 추가
uv add --dev pytest     # 개발 전용 의존성 추가
uv run pytest           # tests/가 존재하면 테스트 실행
```

아직 별도의 빌드 단계는 없습니다. 패키징을 도입하게 되면 선택한 빌드 명령어를 이 파일과 `README.md` 양쪽에 모두 문서화하세요.


## 코딩 스타일 및 이름 짓기 규칙

네 칸 들여쓰기를 사용하는 PEP 8을 따르세요. 모듈, 함수, 변수에는 `snake_case`를 사용하고, 클래스에는 `PascalCase`를 사용하며, 상수에는 `UPPER_SNAKE_CASE`를 사용하세요. 공개 함수와 데이터 구조가 명확하지 않은 로직에는 타입 힌트를 추가하세요. `main.py`를 계속 확장하기보다는 작고 목적이 분명한 모듈을 선호하세요.

현재 포매터나 린터는 설정되어 있지 않습니다. 포매터나 린터를 추가한다면 `pyproject.toml`에 저장소 전체에 적용되는 설정을 두는 것을 권장하며, 변경 사항을 제출하기 전에 실행하세요.

## 테스트 지침

새 테스트에는 `pytest`를 사용하고, 파일 이름은 `tests/test_<module>.py` 형식으로 작성하세요. 테스트 이름은 동작을 설명해야 합니다. 예를 들어 `test_main_prints_greeting`과 같이 작성할 수 있습니다. 성공적으로 실행되는 경우뿐 아니라 새로운 분기와 실패 사례도 테스트하세요. fixture는 작고 결정적으로 유지하세요. 기본 테스트 모음에서 네트워크 접근이나 대용량 모델 다운로드가 필요하지 않도록 하세요.

## 커밋 및 Pull Request 지침

이 저장소에는 커밋 이력이 없으므로 기존 관례를 추론할 수 없습니다. `Add dataset loader` 또는 `Validate training configuration`과 같이 짧고 명령형인 제목을 사용하세요. 커밋은 하나의 목적에 집중되도록 유지하세요.

Pull Request는 목적을 설명하고, 동작 변경 사항을 요약하며, 검증 명령어를 나열하고, 관련 이슈를 연결해야 합니다. 명령줄 동작이나 시각적 동작이 변경되는 경우 샘플 출력이나 스크린샷을 포함하세요. 인증 정보, `.venv/`, 생성된 체크포인트, 대용량 로컬 데이터셋은 절대 커밋하지 마세요.
